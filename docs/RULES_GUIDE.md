## 1. 设计理念

### 核心原则

这套分类体系建立在一个基本假设上：**招聘团队找简历的方式是"我要看产品经理"，而不是"我要看互联网行业"**。

因此，规则采用**职能导向**而非行业导向的分类体系：

| 分类方式 | 问题 |
|---------|------|
| 按行业分（互联网/医疗/金融） | 同一候选人的简历可能适配多个行业，造成重复 |
| 按职级分（初/中/高级） | 职级主观且难从文件名判断 |
| **按职能分（技术/产品/设计）** | ✅ 候选人的核心能力稳定，找人时直接对应 |

### 两层架构

```
第一层：横切叠加标签（OVERLAY TAGS）
  ├── ★留学生/海归  → 适配国际化岗位的优先池
  ├── ★博士/博士后  → 高学历研究型岗位的优先池
  └── ★高级管理     → C级/VP的高管优先池

第二层：主分类（CATEGORY）
  ├── 01_产品经理
  ├── 02_技术研发
  ├── ...（21个主分类）
  └── 99_待人工归类（兜底）
```

**两层不互斥**：一份简历可以同时命中叠加标签和主分类。例如：
- "CFO-Harvard-海归" → 主分类：`15_财务` + 标签：`★留学生/海归` + `★高级管理`

## 3. 叠加标签（Overlay Tags）

### 概念说明

叠加标签是**横跨所有主分类的第一优先级标记**，用于构建特殊候选人优先池。与主分类的关键区别：

| 特性 | 主分类 | 叠加标签 |
|-----|--------|---------|
| 互斥性 | ✅ 互斥（命中即停） | ❌ 不互斥（全部命中） |
| 数量 | 每份简历最多 1 个 | 每份简历 0~N 个 |
| 用途 | 决定放入哪个文件夹 | 决定是否同时放入优先池 |

### 当前叠加标签

#### TAG_OVERSEA — 留学生/海归

**触发条件**：文件名包含以下任意词：

```
中文：留子、留学、海归、留学生、留学回国、海外学历、境外高校、QS百强
英文院校：Oxford, Cambridge, Harvard, MIT, Stanford, Yale, Columbia,
         NYU, UCLA, LSE, UCL, NUS, NTU, HKU, CUHK, HKUST,
         Toronto, Melbourne, Sydney, Waseda, Keio
```

**使用场景**：岗位要求英语或有海外市场需求时，可先从 `★留学生/海归` 池筛选。

---

#### TAG_PHD — 博士/博士后

**触发条件**：

```
博士、PhD、Ph.D、博士后、PostDoc、Postdoctoral、博士学位、Doctor of、D.Phil、博士研究生
```

**使用场景**：科研、算法、高校合作类岗位，优先查看此标签候选人。

---

#### TAG_EXEC — 高级管理

**触发条件**：C 级头衔、VP、副总裁及以上：

```
CEO, CTO, COO, CFO, CMO, CPO, CRO, CHRO,
首席执行官, 首席技术官, 首席运营官, 首席财务官,
首席营销官, 首席产品官,
VP, Vice President, 副总裁,
总裁, 董事长, 联合创始人, 创始人, 合伙人, 执行董事
```

**使用场景**：高管级岗位或董事会级别猎头任务。

## 4. 主分类详解

### 几个容易混淆的分类边界

#### 产品 vs 运营 vs 新媒体

```
产品经理 (01) ← "产品经理"、"产品策略"、"游戏策划"
运营     (07) ← "用户运营"、"电商运营"、"增长运营"（不含内容方向）
新媒体   (08) ← "新媒体运营"、"短视频"、"文案"、"视频剪辑"
```

> ⚠️ **边界处理**：`新媒体运营` 归 08，不归 07。`产品运营` 中若含 `产品经理` 字样则归 01，否则归 07。

---

#### 技术研发 vs AI与算法 vs 硬件

```
技术研发 (02) ← 软件开发（前端/后端/全栈）、测试、运维、DevOps
AI与算法  (03) ← 算法工程师、NLP/CV、大模型、数据科学家
硬件与电子 (04) ← 芯片、PCB、机械、自动驾驶、通信射频
```

> ⚠️ **边界处理**：`嵌入式开发` 归 02（软件属性为主），`FPGA/IC设计` 归 04（硬件属性为主）。

---

#### 市场 vs 销售 vs 运营

```
市场与品牌 (09) ← 品牌推广、公关、整合营销、SEO/SEM（不直接产生销售额）
销售与BD   (10) ← 销售经理、大客户、商务拓展（直接目标是签单）
运营       (07) ← 电商运营、平台运营（兼顾增长和日常运转）
```

---

#### 财务 vs 金融

```
财务 (15) ← 公司内部财务职能（会计、财务分析、审计、税务）
金融 (19) ← 金融行业本身的岗位（投资、基金、银行、PE/VC）
```

> CFO 归 15（财务职能），投资经理归 19（金融行业）。

---

### 各类关键词设计逻辑

每个分类的关键词遵循 **"精确词 → 通用词"** 排序原则：

```python
# ✅ 正确：精确词在前，避免被通用词"抢先"匹配
("07_运营", [
    "用户运营",     # 精确
    "电商运营",     # 精确
    ...
    "运营经理",     # 通用
    "运营总监",     # 通用
])

# ❌ 错误：通用词在前，会导致"新媒体运营"被误归到 07
("07_运营", [
    "运营",        # 太通用，会匹配"新媒体运营"
    "用户运营",
    ...
])
```

## 5. 匹配算法

### 完整流程

```
输入：简历文件名（如 "AI产品经理-张三-PhD-Harvard.pdf"）

步骤一：预处理
  ├── 取文件名（不含路径）
  └── 转为小写（case_insensitive 模式）

步骤二：叠加标签匹配（ALL，不互斥）
  ├── 遍历全部 overlay_tags
  ├── 对每个 tag 的 keywords 逐一检查 `kw in filename`
  └── 所有命中的 tag 均加入结果列表
  → 结果：tags = ["TAG_PHD", "TAG_OVERSEA"]

步骤三：主分类匹配（FIRST_MATCH，互斥）
  ├── 按 order 排序后依次遍历 categories
  ├── 对每个 category 的 keywords 逐一检查 `kw in filename`
  └── 首个命中即停止
  → 结果：category = "01_产品经理", matched_keyword = "AI产品经理"

步骤四：兜底
  如果步骤三无匹配 → category = "99_待人工归类"

输出：ClassifyResult(
  category = "01_产品经理",
  tags = ["TAG_PHD", "TAG_OVERSEA"],
  matched_keyword = "AI产品经理"
)
```

### 时间复杂度

设分类数为 C，每类平均关键词数为 K，文件数为 N：

- 最优：O(N × K)（首个分类命中）
- 最差：O(N × C × K)（所有分类遍历至兜底）
- 叠加标签：O(N × T × K_t)（T 为标签数，必须全部遍历）

对于当前规模（21 类，每类 ~15 词），处理 1000 份简历约 < 10ms。

## 6. 关键词设计规范

### 何时应该新增关键词

✅ 应该新增：
- 有真实简历文件名中出现，且当前分类错误
- 新兴职位出现（如 `提示工程师`、`AI训练师`）
- 特定行业的通用写法（如 `CRA` = 临床研究协调员）

❌ 不应该新增：
- 过于宽泛导致误匹配（如把 `开发` 加入 02，会匹配 `开发运营`）
- 与已有词重复（如 `产品负责人` 已有，不需再加 `产品主负责人`）
- 内部用语，外部简历文件名中几乎不出现

### 添加新关键词的步骤

```json
// 1. 在对应分类的 keywords 数组末尾追加
{
  "id": "03_AI与算法",
  "keywords": [
    "算法工程师",
    ...
    "新关键词"    // 追加在这里
  ]
}
```

```bash
# 2. 用测试文件验证不会误匹配其他分类
python python/resume_classifier.py "包含新关键词的测试文件名.pdf"

# 3. 运行完整测试
pytest python/tests/ -v
```

### 如何判断优先级（顺序）

**规则**：精确度越高、歧义越少的词放越前。

| 场景 | 正确顺序 | 错误顺序 |
|-----|---------|---------|
| 避免 `销售` 匹配 `销售工程师`（应归技术） | `销售工程师` 先于 `销售` | `销售` 在前 |
| 避免 `设计` 匹配 `系统架构设计`（应归技术） | `UI设计师` 先于 `设计` | `设计` 在前 |

## 7. 扩展与自定义

### 方式一：直接编辑 JSON（推荐）

1. 打开 `rules/classification_rules.json`
2. 在对应分类的 `keywords` 数组追加新词
3. 或新增一个分类对象（注意 `order` 不要与已有重复）
4. 运行 `pytest` 验证

### 方式二：Python 代码中传入自定义规则

```python
from python.resume_classifier import ResumeClassifier

# 方式 A：传入自定义路径
clf = ResumeClassifier(rules_path="/path/to/my_rules.json")

# 方式 B：传入字典（适合动态生成规则）
custom_rules = {
    "config": {"match_mode": "case_insensitive", "match_strategy": "first_match",
                "fallback_category": "99_待人工归类"},
    "overlay_tags": [],
    "categories": [
        {"id": "custom_01", "name": "我的自定义分类", "order": 1,
         "keywords": ["关键词A", "关键词B"]}
    ]
}
clf = ResumeClassifier(rules_dict=custom_rules)
```

### 方式三：合并用户自定义规则（多规则来源）

```python
import json

def merge_rules(base_path: str, user_path: str) -> dict:
    """将用户规则合并到基础规则中（用户关键词追加，不覆盖）。"""
    with open(base_path) as f:
        base = json.load(f)
    with open(user_path) as f:
        user = json.load(f)

    # 用户分类中的关键词追加到已有分类
    base_index = {c["id"]: c for c in base["categories"]}
    for cat in user.get("categories", []):
        if cat["id"] in base_index:
            base_index[cat["id"]]["keywords"].extend(cat.get("keywords", []))
        else:
            base["categories"].append(cat)  # 全新分类直接追加

    return base
```

### 方式四：给 AI 提供分类上下文

如果你正在构建 AI 简历解析系统，可以将规则直接注入 Prompt：

```python
def build_classification_prompt(clf: ResumeClassifier) -> str:
    cats = clf.get_categories()
    lines = ["你是一个简历分类助手，请将简历归入以下分类之一：\n"]
    for cat in cats:
        if cat["id"] == "99_待人工归类":
            continue
        lines.append(f"- {cat['id']}（{cat['name']}）：{cat['description']}")
    lines.append("\n如果无法确定，请返回 99_待人工归类。")
    return "\n".join(lines)
```

| 变化 | v1 | v2 |

| 行政财务法务 | 合并在一个分类 | 拆分为 `14_行政`、`15_财务`、`16_法务合规` |

### 迁移脚本

规则文件中的 `legacy_mapping` 字段记录了旧文件夹名到新分类的映射关系，可用于批量重命名历史简历库：

```python
import json, os, shutil

with open("rules/classification_rules.json") as f:
    rules = json.load(f)

mapping = rules["legacy_mapping"]
base_dir = "/path/to/your/resume/library"

for old_name, new_name in mapping.items():
    old_path = os.path.join(base_dir, old_name)
    new_path = os.path.join(base_dir, new_name)
    if os.path.exists(old_path) and not os.path.exists(new_path):
        shutil.move(old_path, new_path)
        print(f"已迁移: {old_name} → {new_name}")
```

## 9. 常见问题

**Q：文件名中含多个职位词（如"产品经理转技术总监"），会归入哪个？**

A：按 `order` 顺序，`01_产品经理` 先于 `02_技术研发` 匹配，因此归入产品经理。如需改变，可调整分类的 `order` 值或更换关键词顺序。

---

**Q：为什么不直接用 AI 做分类，而要维护关键词规则？**

A：关键词规则有几个优势：
1. **速度**：纯字符串匹配 < 1ms，AI 推理需要几百 ms
2. **可解释**：每次分类都知道是哪个关键词触发的
3. **可控**：规则变更立即生效，不需要重新训练
4. **成本**：大量简历处理无额外费用

推荐做法：**关键词规则作为快速通道，AI 兜底处理关键词无法识别的 `99_待人工归类` 简历**。

---

**Q：英文简历文件名是否支持？**

A：支持。当前规则中已包含大量英文关键词（`NLP Engineer`、`Research Scientist`、`DevOps` 等），且匹配默认大小写不敏感。

---

**Q：如何处理"产品运营"这种模糊职位？**

A：`产品运营` 中若含 `产品经理` 字样（如"产品运营经理兼产品经理"）会归入 01，否则归入 07_运营。如果你的场景中 `产品运营` 应归入产品类，可在 01 的关键词中追加 `产品运营经理` 这个更精确的词。

---

**Q：如何为 AI Agent 提供这套规则作为工具/知识？**

A：

```python
# 将规则序列化成自然语言，注入到 System Prompt
clf = ResumeClassifier()
categories_text = "\n".join([
    f"- {c['id']}：{c['name']}（{c['description']}）"
    for c in clf.get_categories()
])

system_prompt = f"""
你是一个简历分类助手。请根据简历的文件名或职位信息，
将候选人归入以下分类之一：

{categories_text}

如果信息不足，请返回 99_待人工归类。
"""
```

## 10. 贡献指南

欢迎为规则库贡献力量！

### 贡献流程

1. Fork 本仓库
2. 在 `rules/classification_rules.json` 中修改或新增关键词
3. 在 `python/tests/test_classifier.py` 中添加对应测试用例
4. 提交 PR，标题格式：`feat(rules): 在 XX 分类中添加 YY 关键词`

### PR 检查清单

- [ ] 新关键词在真实简历文件名中有据可查
- [ ] 测试用例覆盖新关键词的命中和边界情况
- [ ] `pytest python/tests/ -v` 全部通过
- [ ] 如果影响边界（如某个词从 A 类移到 B 类），说明原因

### 提 Issue 的场景

- 某类职位的简历被误分类
- 发现关键词冲突（同一个词在两个分类都出现）
- 需要新增尚未覆盖的职能分类
- 英文职位名称缺失
